# 介護タクシー管理アプリ UI土台 v1

このZIPは、PDFモックを基準に「必要部分だけを綺麗にトレースし、各エリアを個別ファイルで調整できる」ことを目的に作った初版です。

この段階では、完成版の業務アプリではなく、**崩れない画面構造・各エリア責務・LOCATIONスクロール設計を確認するための土台**です。

---

## 重要な前提

このアプリは通常のWebページではありません。

一般的なWebページのように、上から下へコンテンツを積み上げてページ全体をスクロールさせる構造にはしません。

このアプリは、iPadやPCで使う「業務端末UI」です。

そのため、基本思想は以下です。

- 画面全体は固定
- 青帯より下も固定
- 画面全体スクロールは禁止
- 各エリアは固定配置
- スクロールが必要な場合は、そのエリアの中だけで処理
- LOCATIONを主役にする
- DETAIL、DAYS、STATUSは補助エリアとして扱う
- 余計なラフ画像や雰囲気優先の改変は不要
- PDFの位置関係、余白、バランスを基準にする

---

## 今回の目的

このコードは、現状コードを細かく延命修正するためのものではありません。

現状コードは参考にはしますが、根本的には以下の問題がありました。

- 画面全体をscaleで拡大縮小していた
- 固定したいエリアとスクロールさせたいエリアが同じ親要素の影響を受けていた
- LOCATION内部の縦横スクロールが複雑に絡んでいた
- Safari等で帯より下が数px動く可能性があった
- タイムライン線、担当者列、時間軸、予約帯の責務が混ざっていた

今回のv1では、上記を避けるため、**画面全体固定＋各エリア分離＋LOCATION専用スクロール**で作り直しています。

## UI思想

### 1. 画面全体は固定

画面全体をスクロールさせません。

青帯より下が若干動く問題を避けるため、body/html/app-rootを固定し、overflowをhiddenにしています。

### 2. scale()で全体を拡大縮小しない

旧コードで問題になっていたscale前提を使いません。

理由：

- スクロール量と見た目がズレる
- 線の太さがズレる
- タッチ移動量がズレる
- Safariで微妙な揺れが出る

今回のコードは、実寸レイアウトをCSS変数で調整する思想です。

### 3. PDFの余白・位置関係を優先

PDFモックは、単なる見た目ではなく、余白・密度・視線誘導まで含んだ設計資料として扱います。

ただし、赤破線や説明矢印などのガイド要素は実装しません。

### 4. 各エリアは独立ファイルで調整可能

LOCATIONを調整する時にDAYSやDETAILを壊さないことを重視しています。

今後の拡張でも、1つの巨大ファイルへ詰め込みません。

## エリア設計

### HEADER

役割：

- MENU
- 現在時刻
- 予約ボタン

思想：

- 高さ固定
- 画面幅いっぱい
- 余計な情報を置かない
- 下の業務エリアと明確に分ける

---

### LOCATION

主役エリアです。

役割：

- 当日の予約状況
- 業者別タイムライン
- 現在時刻の把握
- 予約帯の選択

#### 基本仕様

- デフォルト表示は5業者
- 6業者以上はLOCATION内部で縦スクロール
- 時間範囲は6:30〜20:00
- 予約登録時点で開始時間は30分単位
- 30分ごとの線を表示
- 00分線は少し強め
- 30分線は弱め
- 現在時刻線は細めの赤実線
- 現在時刻線は分単位で更新

#### 予約帯

表示ルール：

- 上段左寄せ：患者名
- 下段右寄せ：状態
- 状態文字は5文字以下で設定

仮予約・簡易予約：

- 白背景
- 赤枠
- 患者名は黒文字
- 担当者決定時は黒文字
- 未調整時は赤文字

#### 重複予約

同一担当者の時間重複は基本的に作らない設計です。

調整時に被らないように設計するため、タイムライン上で重複を上下2段にして表現する前提ではありません。

## LOCATIONスクロール設計

ここは最重要です。

### 縦横同時スクロールは禁止

指が動き始めた方向でロックします。

- 横方向に動き始めたら横だけ
- 縦方向に動き始めたら縦だけ
- 途中で斜め移動に切り替えない

### 横スクロール

目的：

- 15:00以降や20:00までの予定を見る

仕様：

- 横スクロールは時間軸と予約帯が連動
- 担当者列は横方向に固定
- 行きすぎ防止のため、慣性を強くしすぎない
- 3秒無操作で現在時刻付近へ戻る

### 縦スクロール

目的：

- 6業者以上を確認する

仕様：

- 時間軸は固定
- 担当者列と予約行は縦方向に同期して動く
- 担当者列は横位置としては固定
- 予約帯だけが単独でズレないようにする

注意：

「担当者列を固定」という表現は、横スクロール時に左へ流れないという意味です。
6業者以上を表示するためには、縦方向では担当者名と予約行を同期して動かす必要があります。
これを同期しないと、担当者名と予約帯が一致しなくなるため不整合になります。

## 現在時刻線の設計

### 基本

赤線はタイムライン座標上に存在します。

そのため、横スクロールで現在時刻が画面外に行けば、赤線も画面外に消えます。

これは不整合ではありません。

### 自動復帰

横スクロール後、3秒無操作で現在時刻付近へ戻します。

戻り方は「赤線を固定30%に置く」ではなく、

- 現在時刻より前を約30%
- 現在時刻より後を約70%

見せる思想です。

つまり「今」より「これから」を広めに見せます。

### 端の処理

- 6:30付近では開始端を尊重
- 20:00付近では終了端を尊重
- 無理に余白を作らない

## DAYS設計

### 基本

DAYSは1週間7日表示です。

2週間14日表示ではありません。

理由：

- 1カードを大きく使える
- タップしやすい
- 7行思想と相性が良い
- iPadで見やすい

### DAYSカード内の行構成

基本は7行です。

```text
1. 日付
2. 空行
3. 案件数
4. 調整数
5. 通知枠1
6. 通知枠2
7. 通知枠3
```

高さ的に厳しい場合は、通知枠を減らして構いません。

優先順位：

```text
日付 ＞ 案件数 ＞ 調整数 ＞ 通知枠
```

通知が多い場合、カード内へ無理に出し切りません。

7行を超える通知は、DAYSカードタップ時にDETAILへ表示します。

### DAYSカード色

スクショから拾った方針：

```text
平日       背景 #FFFFFF   枠 #C9CAFF
当日       背景 #CDE9EE   枠 #45C5E3
日曜祝祭日 背景 #F4DEEB   枠 #FF75A3
```

### 通知バッジ

「請求書発行」「支払い」等を直接コードへ固定しません。

通知は状態オブジェクトとして扱います。

今後、設定画面で以下を変更できる前提です。

- 文言
- 表示ON/OFF
- 並び順
- 色

色は自由入力ではなく、選択肢から選ぶ仕様です。

## DETAIL設計

DETAILは情報表示専用です。

ここに編集フォームや操作ボタンを詰め込みません。

### 表示対象

- 予約帯をタップ → 案件詳細
- DAYSをタップ → その日付の詳細
- STATUSをタップ → 業者状態
- 通知をタップ → 通知詳細（将来）

### 案件詳細の基本表示順

```text
1. 時間
2. 患者名
3. 出発 → 到着
4. 担当
5. 状態
6. 特記
```

将来的には、設定画面で表示順変更も視野に入れます。

---

## STATUS設計

STATUSは軽量表示です。

役割：

- 当日協力してくれる業者の状態を一覧で見る

思想：

- LOCATIONと同順
- 当日協力業者の並びは別途設計
- 編集や詳細入力はしない
- 状態テキストは6文字程度に収める
- STATUSをタップしたらDETAILへ表示

## 実装上の注意

### 1. 1ファイルに詰め込まない

各エリアは個別ファイルで調整します。

LOCATIONを触る時にDAYSやDETAILを壊さない構成を守ります。

### 2. コードは全文差し替え前提

部分修正や差分ではなく、ファイル単位で全選択コピペできる形を基本にします。

### 3. ラフ画像生成は不要

今後このプロジェクトでは、UIラフ画像を勝手に生成しません。

必要なのは以下です。

- 設計
- 不整合確認
- 配置ルール
- 実装順
- コード

### 4. PDFモックを基準にする

見た目の雰囲気ではなく、PDFの位置関係・余白・エリア比率を基準にします。

### 5. 不明点は勝手に補完しない

添付資料で読み取れない箇所、不整合がある箇所は、先に確認します。
